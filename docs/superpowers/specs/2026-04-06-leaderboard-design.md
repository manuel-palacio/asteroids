# Leaderboard Design — Google Play Games Services

**Date:** 2026-04-06
**Scope:** Android only
**Approach:** Platform interface injected into `AsteroidsGame` (Option A)

---

## Goals

- Submit the player's final score to a global leaderboard at game over.
- Allow the player to view the global leaderboard via a button on the Game Over screen.
- Keep `core` free of Android imports.
- Sign in to Google Play on demand (when the player taps LEADERBOARD), not at launch.

## Architecture

Five touch points:

| File | Change |
|------|--------|
| `core/GameServices.kt` | New interface: `submitScore(score: Int)`, `showLeaderboard()` |
| `core/AsteroidsGame.kt` | Add `var gameServices: GameServices? = null` |
| `core/screen/GameScreen.kt` | Call `game.gameServices?.submitScore(world.score)` before transitioning to `GameOverScreen` |
| `core/screen/GameOverScreen.kt` | Add LEADERBOARD button (only rendered when `gameServices != null`); constrain retry tap zone |
| `android/GooglePlayServices.kt` | New class implementing `GameServices` using GPGS v2 |
| `android/AndroidLauncher.kt` | Instantiate `GooglePlayServices`, assign to `game.gameServices` |

The desktop launcher requires no changes — `gameServices` stays `null` and all calls are no-ops.

## GameServices Interface

```kotlin
// core/src/main/kotlin/com/palacesoft/asteroids/GameServices.kt
interface GameServices {
    fun submitScore(score: Int)
    fun showLeaderboard()
}
```

## Score Submission

- Called once in `GameScreen.render()` at the moment `world.gameOver` becomes true, before `game.setScreen(GameOverScreen(...))`.
- Fire-and-forget: GPGS queues the score locally and submits when the player is signed in.
- No retry logic needed — GPGS handles offline queuing internally.

## GameOverScreen UI

```
GAME OVER
SCORE  1234

PRESS SPACE OR TAP TO RETRY    ← tap zone: upper 2/3 of screen
LEADERBOARD                    ← tap zone: lower 1/3, only shown if gameServices != null
```

- Tap detection uses `Gdx.input.justTouched()` with a y-coordinate split to distinguish retry vs leaderboard.
- Both zones active on the same frame is not possible (single touch event consumed by whichever zone matches first).

## GooglePlayServices Sign-in Flow

When the player taps LEADERBOARD:

1. Call `PlayGames.getGamesSignInClient(activity).isAuthenticated`
2. **Authenticated** → show leaderboard intent via `PlayGames.getLeaderboardsClient(activity).getLeaderboardIntent(leaderboardId)`
3. **Not authenticated** → call `signIn()`, on success go to step 2, on failure show `Toast` with "Sign in to Google Play to view leaderboard"

Score submission from `GameScreen` is always attempted regardless of auth state. GPGS caches it.

## Configuration

- `android/src/main/res/values/games_ids.xml` — generated by Play Console, contains leaderboard ID string resource.
- Placeholder value `"YOUR_LEADERBOARD_ID"` until Play Console app is registered.
- `AndroidManifest.xml` requires `<meta-data>` tag for `com.google.android.gms.games.APP_ID`.

## Dependencies

Add to `android/build.gradle.kts`:

```kotlin
implementation("com.google.android.gms:play-services-games-v2:19.0.0")
```

## Out of Scope

- Achievements
- Friends leaderboard
- iOS / desktop leaderboard
- Score persistence across uninstalls (handled by GPGS cloud)
